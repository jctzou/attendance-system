# 考勤記錄頁面規格書 (Attendance Record Spec)

> **版本**: v1.4
> **狀態**: 已實作 (Implemented)
> **最後更新**: 2026-02-28
> **目標**: 定義「打卡記錄 (Attendance Record)」頁面之月曆檢視與資料編輯互動。

---

## 1. 頁面功能概觀與角色權限

本頁面提供日曆式的出勤記錄檢視，並允許使用者點擊進行修正（補登/修改）。
- **一般員工 (Employee)**: 僅能查看**自己**的出勤紀錄，無法切換員工篩選器。
- **管理層級 (Manager / Super Admin)**: 預設查看自己，可透過上方「員工下拉選單」切換並檢視任何員工的當月打卡細節與修訂歷程。

---

## 2. 介面佈局與資料流 (UI Layout & Data Flow)

### 2.1 頁面結構
- **操作列 (Toolbar)**:
    - 月份選擇器 (YYYY-MM)，並附帶左右快速切換上/下個月之按鈕。
    - 員工選擇器 (僅管理員可見，顯示格式：`姓名 員編`。若員工尚未設定員編，則僅顯示姓名，不得顯示空括號 `()`)。
- **日曆視圖 (Calendar Grid)**: 
    - 採用 RWD 排版。手機版為單列垂直堆疊，桌面版為七列月曆佈局 (`grid-cols-7`)。
        - **大螢幕日曆自動補位**: 桌面版需依照當月第一天為星期幾進行空白底板補位 (`padding-start`)，確保每週的七天能固定對齊該欄位（如週日在第一欄，週六在第七欄）。手機版則預設隱藏這些補位區塊，維持由上往下的緊湊條列。
        - **智慧定位**: 頁面載入完成且包含「今日」紀錄時，系統需自動平滑捲動 (`scrollIntoView`) 並盡可能置中於今日的卡片位置，提升手機版長列表的操作體驗。

### 2.2 日卡片 (Day Card) 視覺呈現規則
每個「日期」為一個獨立的卡片。為了確保手機版長列表排版的視覺穩定性，**卡片在滑鼠懸停 (Hover) 時嚴禁縮放 (`scale`) 放大效果**。
背景色應區分今日與週末，依據員工的「薪資類型」會有不同的展示邏輯：
- **今日 (Today)**: 藍色系高亮 (`bg-blue-50`)，帶有藍色邊框。
- **週末/例假日 (Weekend)**: 淺灰色系 (`bg-slate-100`)，以明顯與平日工作日（白底）區分。Hover 時可加深為 `bg-slate-200` 提供互動感。

| 顯示項目 | 月薪制 (Monthly) | 鐘點制 (Hourly) |
| :--- | :--- | :--- |
| **打卡時間** | 顯示上下班時間 (HH:mm) | 顯示上下班時間 (HH:mm) |
| **異常狀態標籤** | **顯示** (遲到/早退 - 紅橘字) | **依設顯示** (受開關控制) |
| **午休時數** | **顯示** (若 > 0) | **顯示** (e.g., "休 1h") |
| **請假資訊** | 顯示 (中文假別黃標) | 顯示 (中文假別黃標) |
| **工作時數** | 不顯示於卡片表面 | 不顯示於卡片表面 |

## 3. 特殊頁面邏輯與狀態管理 (Local Logic & State)

點擊任何一格日卡片，將啟動下方的 **操作對話框 (`AttendanceActionDialog`)**。

> **設計決策 (v1.3)**：「請假申請」功能已從本對話框中完整移除。員工若需申請請假，請前往 **「請假管理 (`/leaves`)」** 頁面操作。打卡記錄頁的對話框專注於**打卡補登與修改**，不再兼具請假申請流程。

### 3.1 狀態 1: 無任何記錄 (Empty)
- **觸發**: 該日沒有任何打卡紀錄與請假申請。
- **可執行動作**:
  - `[補登打卡]`：從無到有新增上下班時間。

### 3.2 狀態 2: 已有打卡記錄 (Has Attendance)
- **觸發**: 該日已有打卡紀錄。
- **可執行動作**:
  - `[修改打卡]`：變更原本填寫的時間，送出後該筆紀錄會被記上 `is_edited=true` 標記。

### 3.3 狀態 3: 已有請假申請 (Has Leave)
- **觸發**: 該日已有任何狀態的請假紀錄。
- **對話框行為**:
  - 在打卡表單頂部顯示**唯讀請假狀態橫幅**，標示假別（由 `LEAVE_TYPE_MAP` 轉換為中文）與審核狀態（待審核 / 已核准 / 申請取消中）。
  - 依然提供打卡補登或修改功能（用於半天假情境），但**不提供任何請假申請或取消的操作入口**。
  - 若需取消或管理假單，員工需前往「請假管理」頁面操作。

### 3.4 假別顯示轉換 (Localization)
資料庫的 `leave_type` 欄位為英文字元 (如 `sick` / `annual`)，前端讀取後**嚴禁**直接顯示原始值，必須透過內建的 `LEAVE_TYPE_MAP` 轉換為中文（如 `病假` / `特休`），並以顯眼的黃底標籤呈現。

### 3.5 單邊打卡修改支援 (Single-sided Modification)
- **情境**: 員工當日已經上班打卡，但**尚未下班**，卻需要由管理員先行修改其上班時間（例如補登忘記打卡的紀錄）。
- **表單規範**:
  - `AttendanceActionDialog` 中的「下班時間 (clock_out_time)」欄位為**非必填 (Optional)**。
  - 對於鐘點制人員，需提供「清除下班時間」的按鈕機制，允許將誤填的下班時間恢復為空值。
- **計算引擎防呆 (Server-side)**:
  - 當後端 (`updateAttendance`) 接收到修改請求且 `clockOutTime` 為 `null` 時。
  - 系統**僅重算遲到狀態 (Late)**。
  - 系統**必須跳過且不計算**「早退狀態 (Early Leave)」與「當日淨工時 (Work Hours)」。工時欄位將被寫入為 `null`，以便日後員工實際執行下班打卡時，系統能以最新的狀態再次結算。

### 3.6 補登/修改對話框的時間安全規範（Bug Fix v1.4）

#### 日期綁定防呆（`workDate` prop）

`TimeSlotSelector` 必須接受 `workDate` prop 作為強制日期來源：

```
優先層級：workDate（外部傳入）> value 中解析的日期 > 今日（最後 fallback）
```

**問題背景**：使用者在補登/修改對話框中，若按下「清除下班時間」或在 `clockOut` 為空時直接選擇時間，原本的邏輯會 fallback 至今天的日期，導致 `clock_out_time` 被寫入今日（而非補登的工作日期），造成工時計算跨越數天。

**修正**：`AttendanceActionDialog` 開啟時，必須將當日 `workDate`（即 `date` prop）傳入所有 `TimeSlotSelector` 元件，確保不論 `clockOut` 是否為空，選擇時間後的完整 datetime 永遠使用補登的對應日期。

**受影響元件**：
- `components/TimeSlotSelector.tsx`：加入 `workDate?: string` prop
- `components/AttendanceActionDialog.tsx`：上班/下班的 TimeSlotSelector 均需傳入 `workDate={date}`

#### 午休時間選項規範

| 規則 | 說明 |
|------|------|
| **固定選項** | `0h / 1h / 1.5h / 2h`（不可任意擴充）|
| **初始值保留** | 修改模式開啟時，若原本午休有值（非 0），**保留原值不歸零** |
| **智慧防呆觸發時機** | 只在使用者**主動改動上班或下班時間**後才觸發，初始化不觸發 |
| **觸發條件（兩者任一）** | ① 上班時間 ≥ 12:00（午後才上班）；② 下班時間 < 12:00（上午就下班）|
| **觸發結果** | 午休自動歸零為 `0h` |
| **未觸發時** | 午休保留使用者原有或自行選擇的值，不強制修改 |

**實作機制**：使用 `useRef hasUserInteracted` 追蹤交互狀態。對話框初始化時 `hasUserInteracted.current = false`，使用者操作 `TimeSlotSelector` 時設為 `true`。防呆 `useEffect` 會先檢查此 flag，確保「打開對話框時不自動觸發」。
