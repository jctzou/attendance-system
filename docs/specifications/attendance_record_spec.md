# 考勤記錄頁面規格書 (Attendance Record Spec)

> **版本**: v1.6
> **狀態**: 已實作 (Implemented)
> **最後更新**: 2026-03-01
> **目標**: 定義「打卡記錄 (Attendance Record)」頁面之月曆檢視與資料編輯互動。

---

## 1. 頁面功能概觀與角色權限

本頁面提供日曆式的出勤記錄檢視，並允許使用者點擊進行修正（補登/修改）。
- **一般員工 (Employee)**: 僅能查看**自己**的出勤紀錄，無法切換員工篩選器。
- **管理層級 (Manager / Super Admin)**: 預設查看自己，可透過上方「員工下拉選單」切換並檢視任何員工的當月打卡細節與修訂歷程。

---

## 2. 介面佈局與資料流 (UI Layout & Data Flow)

### 2.1 頁面結構
- **操作列 (Toolbar)**:
    - 月份選擇器 (YYYY-MM)，並透過 `useSearchParams` 保持 URL 同步以便書籤或分享。
    - 員工選擇器 (僅管理員可見，顯示格式：`姓名 員編`。若員工尚未設定員編，則僅顯示姓名)。
- **日曆視圖 (Calendar Grid)**: 
    - 採用 RWD 排版。手機版為單列垂直堆疊，桌面版為七列月曆佈局 (`grid-cols-7`)。
        - **大螢幕日曆自動補位**: 桌面版需依照當月第一天為星期幾進行空白底板補位 (`padding-start`)。
        - **智慧定位**: 頁面載入完成後自動平滑捲動 (`scrollIntoView`) 至今日卡片。

### 2.2 日卡片 (Day Card) 視覺呈現規則
每個「日期」為一個獨立的卡片，背景色區分今日與週末。
- **今日 (Today)**: 藍色系高亮 (`bg-blue-50`)，帶有藍色邊框。
- **週末/例假日 (Weekend)**: 淺灰色系 (`bg-slate-100`)。

| 顯示項目 | 月薪制 (Monthly) | 鐘點制 (Hourly) |
| :--- | :--- | :--- |
| **打卡時間** | 顯示上下班時間 (HH:mm) | 顯示上下班時間 (HH:mm) |
| **異常狀態標籤** | **顯示** (遲到/早退 - 紅橘字) | **依設顯示** (受開關控制) |
| **午休時數** | **顯示** (若 > 0) | **顯示** (e.g., "休 1h") |
| **工時精確度 (v1.5)** | 兩位小數 (如 8.33h) | **兩位小數 (如 7.25h)** |
| **工作時數** | 不顯示於卡片表面 | 不顯示於卡片表面 |

## 3. 特殊頁面邏輯與狀態管理 (Local Logic & State)

點擊任何一格日卡片，將啟動下方的 **操作對話框 (`AttendanceActionDialog`)**。

### 3.1 補登與修改流程 (v1.5)
- **TimeSlotSelector 管理**:
    - 工作日期由 `AttendanceActionDialog` 強制傳入 `workDate` prop。
    - 清除時間後重新選擇時，年份/月份/日期不再跳回「今日」，始終維持原設定日期。
- **午休選項**:
    - 預設提供 `0h / 1h / 1.5h` 之按鈕選項，支援彈性調整。
    - **防呆連鎖**: 若上班晚於 12:00 或下班早於 12:00（且使用者有主動交互 `hasUserInteracted`），自動將午休歸零為 `0h`。

## 4. 前端效能優化與狀態管理 (Performance Optimization)

為了處理每月高達 31 天的日曆渲染與龐大資料庫查詢（出勤與請假），系統對打卡紀錄頁面 (`app/attendance/page.tsx`) 採取了三次元的深度優化：

### 4.1 雙管齊下的非同步載入 (Promise.all)
- **問題**：原系統採用序列化請求（等待出勤回傳後，才請求請假資料），導致畫面出現瀑布式的等待延遲 (Waterfall Requests)。
- **解法**：全面改用 `Promise.all` 並發請求，大幅縮短 Server Action 來回時長，等待載入的時間砍半。

### 4.2 O(1) 字典查表法快取 (useMemo Hash Maps)
- **問題**：要在畫面上生成 31 張卡片 (`DayCard`)，每一張卡片都會在原始陣列中執行高耗能的 `.find()` 與時間物件 (`Date`) 轉換。若單月請假過多，畫面刷新將重複執行上百次迴圈。
- **解法**：在 API 資料拿到的瞬間，透過 `useMemo` 將陣列轉換為以日期為 Key 的物件字典 (`attendanceMap` 與 `leaveMap`)。渲染每一天的卡片時，只需執行時間複雜度為 O(1) 的尋址操作（例如 `attendanceMap['2026-03-04']`），完全消滅了暴力查詢。這為未來若要在 `DayCard` 加入複雜工時公式打下完美的效能基礎。

### 4.3 元件記憶化防護 (React.memo)
- **問題**：用戶切換月份、修改單日資料或介面重繪時，React 會無差別地把 31 張卡片全部拆掉重畫。
- **解法**：底層卡片元件 `DayCard.tsx` 已全面封裝 `React.memo`。只要這天的參數（出勤資料與請假資料）沒有改變，系統就會直接沿用繪製好的快取圖層。這使操作回饋變得如原生 App 般順滑。

### 4.4 管理員全域快取與接縫切換 (Global Cache & Zero-Latency)
- **問題**：管理員透過下拉選單切換查看不同員工的考勤時，即使是同一個月份，原本也必須面臨「等待請求 -> 取得員工 A 資料 -> 伺服器回傳 -> 重新渲染」的數秒延遲。
- **解法**：建立管理員專屬大水管（`getAllEmployeesMonthlyAttendance`, `getAllEmployeesMonthlyLeaves`）。當確認登入者為管理員時，系統會一次性將「當月全公司的打卡及請假資料」載入前端。配合前述的 2D 字典查表法，當下拉選單切換 `selectedEmployee` 時，系統不再發送網路請求，而是直接從本機的 `attendanceMap` 重新指派給對應介面，達成完美的多員工切換體驗。

### 4.5 全系統共用載入元件 (Skeleton & Spinner)
- 在等待 Server Action 執行的期間（如初次進入頁面或大規模資料抓取時），系統使用共用的 `LoadingSpinner.tsx`（動態旋轉圖示）與 `DayCardSkeleton.tsx`（骨架屏模組）。畫面會預先繪製出帶有呼吸閃爍效果 (`animate-pulse`) 的骨架卡片。這在心理學上有效降低了等待的僵硬感，打造出接近原生 App 的現代化流暢質感。
