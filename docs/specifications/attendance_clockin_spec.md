# 打卡系統設計規格書 (Attendance Clock-in Spec)

> **版本**: v2.2
> **狀態**: 已實作 (Implemented)
> **最後更新**: 2026-03-02
> **目標**: 建立直覺、防呆的員工打卡介面，統一上班邏輯，優化鐘點下班彈性。

---

## 1. 頁面功能概觀與角色權限

本頁面為所有 **員工 (Employee/Manager)** 每日使用的核心功能。
負責即時記錄上下班時間，並立即回饋當天的出勤狀態（正常/遲到/早退）。

---

## 2. 介面佈局與資料流 (UI Layout & Data Flow)

### 2.1 共通元件 (Common Components)
介面以「現代化卡片 (Modern Card)」為核心，位於畫面正中央。
- **時鐘區**: 顯示伺服器與客戶端同步的實時標準時間，採用超大字級 (`text-6xl`) 確保清晰。
- **狀態顯示**: 
    - 未上班：顯示「尚未打卡」。
    - 上班中：顯示今日上班時間、午休選擇、目標下班時間。
    - 已下班：顯示總結卡片（含上下班時間、總淨工時、扣除午休時數）。其工時數值忠實呈現系統計算結果，精確至小數點後二位 (v2.1)。

### 2.2 鐘點制與月薪制的營運差異 (v1.9 更新)

- **上班打卡 (統一)**:
    - 不分班別，點擊上班即記錄 **當下系統時間**。 
    - 移除原本鐘點制的 30 分鐘級距選擇 (Legacy Calibration 棄用)。
    - **遲到判定緩衝 (v2.2 特別更新)**: 月薪人員享有 **5 分鐘** 的上班緩衝（不計遲到）；鐘點人員則維持精確判定（無緩衝）。
- **下班打卡 (月薪制更新)**:
    - **結算畫面**: 與鐘點制統一格式，顯示超大工時數字，下方顯示精確扣除的午休時數 (v1.9)。
    - **午休取值**: 優先讀取 `users.break_hours` 設定。若未設定則預設為 1.0 小時。
- **下班打卡 (鐘點制特化)**:
    - 下班時間 **不採用** 當下點擊時間，而是採用「表定下班時間」或「調整後的下班時間」。
    - **表定時間**: 自動載入 `users.work_end_time` (預設 18:00)。
    - **調整機制**: 點擊「調整時間」可開啟時段選擇器。
        - **智慧比對**: 系統動態比對選擇時間與表定時間 (HH:mm)。
        - **標籤恢復邏輯**: 
            - 若選擇時間 **不等於** 表定時間（或重新整理後帶入非表定時間）：顯示標籤「下班時間 (已調整)」，並使用橘色標註。
            - 若選擇時間 **調回等於** 表定時間：標籤自動恢復為「表定下班時間」，並移除所有調整標記與顏色，確保顯示狀態隨資料真實性即時變動。
        - **按鈕文字**: 統一固定為「調整時間」，不論調整與否皆不變動文字。
        - **提示文字**: 在下班與午休區塊間顯示「若有特殊狀況，可點『調整』修改下班時間」。
    - **午休選擇**: 提供 `0 hr`, `1.0 hr (預設)`, `1.5 hr` 三個快速按鈕。
- **流程體驗**:
    - 全程採用 **樂觀更新 (Optimistic UI)**。
    - 點擊打卡或取消下班後 **無彈窗/無 Toast 提示**，直接流暢切換介面狀態視圖（月薪制成功後仍保留綠色打勾提示）。

## 3. 特殊頁面邏輯與狀態管理 (Local Logic & State)

### 3.1 頁面視圖切換 (View States)
- **NOT_CLOCKED_IN**: 僅顯示「上班打卡」大按鈕。
- **WORKING**: 顯示上班時間、下班預計（可調整）、午休。點擊「下班打卡」直接結算。
- **CLOCKED_OUT**: 顯示統計結果與「取消下班」微型連結。

### 3.2 誤按防呆：取消下班 (Cancel Clock Out)
- 點擊「取消下班」後，經確認直接返回「上班中 (WORKING)」狀態。
- **狀態重置**: 取消後將「調整標記」清除，回復為預設表定時間。

### 3.3 計算引擎規則
- **工時計算**: 統一採用 `utils/attendance-engine.ts`，四捨五入至小數第二位。
- **棄用機制**: 正式移除舊有的「鐘點人員工時校正至 0.5 整數」邏輯，改為精確計算。

### 3.4 出勤狀態判定規則 (Attendance Status Rules)
- **月薪人員 (Monthly Workers)**:
    - **上班緩衝**: 提供 **5 分鐘** 的遲到緩衝。若在表定上班時間後的 5 分鐘內 (含) 完成打卡，系統判定為「正常 (Normal)」。超過 5 分鐘則標註為「遲到 (Late)」。
- **鐘點人員 (Hourly Workers)**:
    - **無狀態判定**: 系統 **不進行** 遲到或早退判定。只要有打卡記錄，其狀態一律寫入為「正常 (Normal)」。
    - **目的**: 鐘點制採實到實付，工時已精確記錄，無需遲到/早退之行政處分標籤。
- **早退判定**: 針對月薪人員，於表定下班時間前完成打卡，皆標註為「早退 (Early Leave)」。

## 4. 補登與修改打卡規則 (Backfill & Edit Rules)

針對手動「補登打卡」或「修改打卡記錄」的彈窗介面，設定以下時間選擇規則：

### 4.1 鐘點人員 (Hourly Workers)
為了在維持工作記錄精確性的同時，確保符合營運慣例：
- **上班時間**: 
    - **不限分鐘數**：可精確設定到每一分鐘 (e.g., 09:13, 10:47)。
    - **目的**：真實反映員工到達並開始工作的準確時間。
- **下班時間**:
    - **以 30 分鐘為單位**：僅能選擇整點或半點 (e.g., 18:00, 18:30)。
    - **目的**：維持簡潔的結算級距，符合公司對於下班工時結算的標準化流程。

### 4.2 修改原因紀錄
- 任何手動補登或修改皆必須填寫原因。
- 系統會自動記錄修改歷程，並儲存於 `attendance_edit_logs` 以供稽核。

 ## 5. AI 運勢導師功能 (AI Fortune Integration)
 
 為了提升員工入職時的正向心理感受，系統整合了 AI 運勢導師功能。
 
 ### 5.1 運作邏輯
 - **顯示觸發**: 僅在「上班中 (WORKING)」狀態下顯示，給予員工上班期間的心理建設與鼓勵。
 - **暫存策略**: 採本地 **「同一自然日 (Taipei Date)」** 暫存。
     - 頁面載入時，若本地已有當日運勢內容，則不再重複調用 API。
     - 跨日 (00:00) 後首次進入頁面會自動失效並更新。
 - **手動更新**: 點擊運勢區塊可強制忽略暫存，重新向 AI 抓取最新的內容。
 - **後端技術**: 採用 **OpenAI API (GPT-4o-mini)** 進行即時分析。
 - **服務入口**: 由 `app/attendance/fortune.ts` 封裝，透過 Server Action 提供服務。
  - **觸發時機**: 頁面加載後，若偵測為「上班中」且目前無資訊，則自動發送請求。
 
 ### 5.2 內容規範 (Prompt 策略)
 - **角色定位**: niizo 工作室專屬命理師。
 - **分析內容**: 結合當日日期與姓名，隨機選擇命理依據（如：天氣、八字、西洋占星等），並結合工作日常。
 - **結構限制**: 
     - 總字數嚴格限制在 **50 字內**。
     - 包含一句溫度的提示與一個幸運物/幸運色。
 - **語言風格**: 直接輸出繁體中文，語氣親切、幽默，符合職人文化。
 
 ### 5.3 錯誤與備援 (Fallback)
 - 若 API 被限流、Key 失效或網路異常，系統會自動 fallback 顯示預設正向語句：「今天也是充滿希望的一天，加油！」，確保使用者體驗不中斷。
