# 打卡系統設計規格書 (Attendance Clock-in Spec)

> **版本**: v2.3
> **狀態**: 已實作 (Implemented)
> **最後更新**: 2026-03-03
> **目標**: 建立直覺、防呆的員工打卡介面，統一上班邏輯，鐘點下班全面回歸實時打卡。

---

## 1. 頁面功能概觀與角色權限

本頁面為所有 **員工 (Employee/Manager)** 每日使用的核心功能。
負責即時記錄上下班時間，並立即回饋當天的出勤狀態（正常/遲到/早退）。

---

## 2. 介面佈局與資料流 (UI Layout & Data Flow)

### 2.1 共通元件 (Common Components)
介面以「現代化卡片 (Modern Card)」為核心，位於畫面正中央。
- **時鐘區**: 顯示伺服器與客戶端同步的實時標準時間，採用超大字級 (`text-6xl`) 確保清晰。
- **狀態顯示**: 
    - 未上班：顯示「尚未打卡」。
    - 上班中：顯示今日上班時間、午休選擇。
    - 已下班：顯示總結卡片（含上下班時間、總淨工時、扣除午休時數）。其工時顯示為「x小時y分」格式，底層精確記錄每一分鐘 (v2.3)。下班時會自動比對「實收工時」與「表定工時」，若誤差達 11 分鐘以上，會出現「比表定時間不足/超過 X 分」之提示。

### 2.2 鐘點制與月薪制的營運差異 (v1.9 更新)

- **上班打卡 (統一)**:
    - 不分班別，點擊上班即記錄 **當下系統時間**。 
    - 移除原本鐘點制的 30 分鐘級距選擇 (Legacy Calibration 棄用)。
    - **遲到判定緩衝 (v2.2 特別更新)**: 月薪人員享有 **5 分鐘** 的上班緩衝（不計遲到）；鐘點人員則維持精確判定（無緩衝）。
- **下班打卡 (月薪制更新)**:
    - **結算畫面**: 與鐘點制統一格式，顯示超大工時數字，下方顯示精確扣除的午休時數 (v1.9)。
    - **午休取值**: 優先讀取 `users.break_hours` 設定。若未設定則預設為 1.0 小時。
- **下班打卡 (鐘點制更新)**:
    - **規則回歸**: 下班時間正式回歸採用 **當下系統時間**，與上班邏輯一致。
    - **移除舊規**: 移除原本「顯示/調整表定下班時間」的所有相關功能、介面與提示。
    - **午休選擇**: 提供 `0 hr`, `1.0 hr (預設)`, `1.5 hr` 三個快速按鈕。
- **流程體驗**:
    - 全程採用 **樂觀更新 (Optimistic UI)**。
    - 點擊打卡或取消下班後 **無彈窗/無 Toast 提示**，直接流暢切換介面狀態視圖（月薪制成功後仍保留綠色打勾提示）。

## 3. 特殊頁面邏輯與狀態管理 (Local Logic & State)

### 3.1 頁面視圖切換 (View States)
- **NOT_CLOCKED_IN**: 僅顯示「上班打卡」大按鈕。
- **WORKING**: 顯示上班時間、午休。點擊「下班打卡」直接結算。
- **CLOCKED_OUT**: 顯示統計結果與「取消下班」微型連結。

### 3.2 誤按防呆：取消下班 (Cancel Clock Out)
- 點擊「取消下班」後，經確認直接返回「上班中 (WORKING)」狀態。
- **狀態重置**: 取消後回復為預設數據顯示。

### 3.3 計算引擎規則
- **工時計算**: 統一採用 `utils/attendance-engine.ts`。
- **資料儲存 (v2.3 更新)**: 
    - `attendance.work_minutes`: 儲存為 **分鐘 (Minutes)** 數，採整數記錄以維持計算精確度。
    - `attendance.break_duration`: 儲存為 **分鐘 (Minutes)** 數。
    - **顯示層**: UI 顯示時會自動將分鐘轉換回小時與分鐘格式，例如：**8小時6分**。規則說明：
        - 數字與文字間**無空格**。
        - 分鐘部分簡寫為「**分**」，不使用「分鐘」二字。
- **棄用機制**: 正式移除舊有的「鐘點人員工時校正至 0.5 整數」邏輯，改為精確計算。

### 3.4 出勤狀態與工時差額提示 (Status & Difference Warning)
- **取消行政處分標籤**:
    - **所有人員 (月薪與鐘點)** 系統皆 **不進行** 遲到或早退判定。只要有打卡記錄，其入庫狀態一律強制寫入為「正常 (Normal)」。
    - **目的**: 回歸實收工時制，用客觀的數字說話，避免傳統打卡系統的紅綠標籤壓力。
- **下班差額提示 (v2.3 新增)**:
    - 系統會在下班打卡後，計算**落差分鐘數 (Diff)**：`落差 = (實際淨工時) - (表定應上總工時 - 午休)`
    - **寬容度區間**: 若落差落在 `-10` 到 `+10` 分鐘之間，系統視為正常誤差，不顯示任何提示。
    - **不足提示**: 若落差 `<= -11` 分鐘，顯示紅字「**比表定時間不足 X小時Y分**」（若小於 1 小時則顯示「**比表定時間不足 Y分**」）。
    - **超過提示**: 若落差 `>= 11` 分鐘，顯示藍字「**比表定時間超過 X小時Y分**」（若小於 1 小時則顯示「**比表定時間超過 Y分**」）。

## 4. 補登與修改打卡規則 (Backfill & Edit Rules)

針對手動「補登打卡」或「修改打卡記錄」的彈窗介面，設定以下時間選擇規則：

### 4.1 鐘點人員 (Hourly Workers)
為了在維持工作記錄精確性的同時，確保符合營運慣例：
- **上班時間**: 
    - **不限分鐘數**：可精確設定到每一分鐘 (e.g., 09:13, 10:47)。
    - **目的**：真實反映員工到達並開始工作的準確時間。
- **下班時間**: 
    - **不限分鐘數**：可精確設定到每一分鐘 (e.g., 18:03, 19:12)。
    - **目的**：因應實時打卡制度，補登亦需能反映真精確的離開時間。

### 4.2 修改原因紀錄
- 任何手動補登或修改皆必須填寫原因。
- 系統會自動記錄修改歷程，並儲存於 `attendance_edit_logs` 以供稽核。

 ## 5. AI 運勢導師功能 (AI Fortune Integration)
 
 為了提升員工入職時的正向心理感受，系統整合了 AI 運勢導師功能。
 
 ### 5.1 運作邏輯
 - **顯示觸發**: 僅在「上班中 (WORKING)」狀態下顯示，給予員工上班期間的心理建設與鼓勵。
 - **暫存策略**: 採本地 **「同一自然日 (Taipei Date)」** 暫存。
     - 頁面載入時，若本地已有當日運勢內容，則不再重複調用 API。
     - 跨日 (00:00) 後首次進入頁面會自動失效並更新。
 - **手動更新**: 點擊運勢區塊可強制忽略暫存，重新向 AI 抓取最新的內容。
 - **後端技術**: 採用 **OpenAI API (GPT-4o-mini)** 進行即時分析。
 - **服務入口**: 由 `app/attendance/fortune.ts` 封裝，透過 Server Action 提供服務。
  - **觸發時機**: 頁面加載後，若偵測為「上班中」且目前無資訊，則自動發送請求。
 
 ### 5.2 內容規範 (Prompt 策略)
 - **角色定位**: niizo 工作室專屬生活導師與智慧長輩。
 - **分析內容**: 每次包含兩部分：
     1. 【固定項目：午餐】：開頭說「午餐推薦你吃...」並從特定清單（如麗媽、八方等 20 家）隨機挑選一家。
     2. 【隨機二選一】：從「今日氣象與穿著建議」或「個人健康提醒」中隨機選一項加入。
 - **結構限制**: 
     - 合併輸出成一段話，不顯示分類標籤。
     - 總字數嚴格限制在 **50 字內**。
     - 直接點名員工，不使用具有明顯時間段（早/午/晚）的問候語。
 - **語言風格**: 直接輸出繁體中文，語氣溫暖、沉穩且具備長輩智慧感。
 
 ### 5.3 錯誤與備援 (Fallback)
 - 若 API 被限流、Key 失效或網路異常，系統會自動 fallback 顯示預設正向語句：「今天也是充滿希望的一天，加油！」，確保使用者體驗不中斷。

## 7. 前端效能優化與狀態管理 (Performance Optimization)

為了處理每月高達 31 天的日曆渲染與龐大資料庫查詢（出勤與請假），系統對打卡紀錄頁面 (`app/attendance/page.tsx`) 採取了三次元的深度優化：

### 7.1 雙管齊下的非同步載入 (Promise.all)
- **問題**：原系統採用序列化請求（等待出勤回傳後，才請求請假資料），導致畫面出現瀑布式的等待延遲 (Waterfall Requests)。
- **解法**：全面改用 `Promise.all` 並發請求，大幅縮短 Server Action 來回時長，等待載入的時間砍半。

### 7.2 O(1) 字典查表法快取 (useMemo Hash Maps)
- **問題**：要在畫面上生成 31 張卡片 (`DayCard`)，每一張卡片都會在原始陣列中執行高耗能的 `.find()` 與時間物件 (`Date`) 轉換。若單月請假過多，畫面刷新將重複執行上百次迴圈。
- **解法**：在 API 資料拿到的瞬間，透過 `useMemo` 將陣列轉換為以日期為 Key 的物件字典 (`attendanceMap` 與 `leaveMap`)。渲染每一天的卡片時，只需執行時間複雜度為 O(1) 的尋址操作（例如 `attendanceMap['2026-03-04']`），完全消滅了暴力查詢。這為未來若要在 `DayCard` 加入複雜工時公式打下完美的效能基礎。

### 7.3 元件記憶化防護 (React.memo)
- **問題**：用戶切換月份、修改單日資料或介面重繪時，React 會無差別地把 31 張卡片全部拆掉重畫。
- **解法**：底層卡片元件 `DayCard.tsx` 已全面封裝 `React.memo`。只要這天的參數（出勤資料與請假資料）沒有改變，系統就會直接沿用繪製好的快取圖層。這使操作回饋變得如原生 App 般順滑。

### 7.4 管理員全域快取與 0 秒切換 (Global Cache & Zero-Latency UI)
- **問題**：管理員透過下拉選單切換查看不同員工的考勤時，即使是同一個月份，原本也必須面臨「等待請求 -> 取得員工 A 資料 -> 伺服器回傳 -> 重新渲染」的數秒延遲。
- **解法**：建立管理員專屬大水管（`getAllEmployeesMonthlyAttendance`, `getAllEmployeesMonthlyLeaves`）。當確認登入者為管理員時，系統會一次性將「當月全公司的打卡及請假資料」載入前端。配合前述的 2D 字典查表法，當下拉選單切換 `selectedEmployee` 時，系統不再發送網路請求，而是直接從本機的 `attendanceMap` 重新指派給對應介面，達成完美的多員工「0 秒光速切換」體驗。

### 7.5 全系統共用載入元件 (Skeleton & Spinner)
- **問題**：在等待 Server Action 執行的期間（如初次進入頁面或大規模資料抓取時），單調的「載入中...」文字無法提供良好的視覺回饋，容易讓使用者產生焦慮感。
- **解法**：建立全系統共用的 `LoadingSpinner.tsx`（動態旋轉圖示）與 `Skeleton.tsx`（骨架屏模組）。在打卡紀錄頁中，更特別實作了與真實日曆相仿的 `DayCardSkeleton.tsx`。當網路還在抓取資料時，畫面會預先繪製出 31 張帶有呼吸閃爍效果 (`animate-pulse`) 的骨架卡片。這在心理學上有效降低了等待的僵硬感，打造出接近原生 App 的現代化流暢質感。
